# Daily Digest — Update Obsidian Vault

Generate a human-readable project overview and write it to the Obsidian vault.
This is a dashboard for the human, not documentation for agents.

## What to read

1. `AGENTS.md` — project name, description, "This project is NOT"
2. `git log --oneline -20` — recent activity
3. Latest file in `changelog/session-logs/` — what happened last
4. `docs/decisions/` — list all ADRs, extract title and status only
5. `docs/research/` — note the latest research date
6. Tribal Knowledge section from `AGENTS.md`
7. `CHANGELOG.md` — latest release version and unreleased items

## What to write

Write a single markdown file to:
```
~/workspace/vault/Projects/TODO-PROJECT-NAME.md
```

Use this structure:

```markdown
# Project Name

> One-line description from AGENTS.md.

**Status:** (synthesize from git log + session log — e.g. "Active — working on auth module")
**Version:** (from VERSION file)
**Last session:** (date + one-sentence summary from latest session log)
**Repo:** (GitHub URL from .git/config)

## What it is

(2–3 sentences from AGENTS.md description)

## What it is NOT

(bullet list from AGENTS.md anti-goals)

## Current state

(What's done, what's in progress, what's blocked. Written as prose,
not a copy-paste of logs. The human should understand the situation
in 30 seconds.)

## Key decisions

(Table: ADR number, title, status. One row per ADR. No full text —
the human can click through to the repo if they need details.)

| ADR | Title | Status |
|-----|-------|--------|
| ... | ...   | ...    |

## What's next

(From the latest session log's "next steps" or from open GitHub Issues.
Prioritized, not exhaustive.)

## Tribal Knowledge

(Copy the Tribal Knowledge section from AGENTS.md verbatim —
this is the stuff the human should know too.)

---
*Auto-generated by agent on YYYY-MM-DD. Source: `~/workspace/projects/TODO/`*
```

## Rules

- Overwrite the existing file every time — this is a snapshot, not a history.
- Keep it under 100 lines. Compress ruthlessly. Link instead of copying.
- Write for a human who hasn't looked at this project in two weeks.
- Never include secrets, tokens, passwords, or file contents.
- If something is unclear or missing, write "Unknown — check [location]"
  rather than guessing.
